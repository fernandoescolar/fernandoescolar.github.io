## Google Cloud: k8s-aibom para ML‑BOMs en GKE
Google Cloud ha publicado y abierto el código de `k8s-aibom`, un controlador no privilegiado para Kubernetes que detecta automáticamente runtimes de IA en ejecución (vLLM, Triton, TGI, Ollama, etc.), frameworks de agentes, bases de datos vectoriales y trabajos de entrenamiento. El controlador genera documentos estándar CycloneDX 1.6 en formato ML‑BOM a partir de la observación en tiempo de ejecución, adjunta esos BOMs como estado en un recurso personalizado y puede exportarlos a sinks externos (Cloud Storage, webhooks). El diseño prioriza baja fricción (sin sidecars ni DaemonSets privilegiados), trazabilidad determinista y un modelo de confianza (declared / inferred / unresolved) pensado para auditoría y cumplimiento. [Google Cloud: Introducing k8s-aibom on GKE for automated AI bills of materials](https://cloud.google.com/blog/products/identity-security/introducing-k8s-aibom-on-gke-for-automated-ai-bills-of-materials). ([cloud.google.com](https://cloud.google.com/blog/products/identity-security/introducing-k8s-aibom-on-gke-for-automated-ai-bills-of-materials?utm_source=openai))

## GitHub: estimador de coste de licencias de Code Quality (vista previa)
GitHub añadió una tarjeta en la página de facturación que ofrece una estimación del número de committers activos usados por GitHub Code Quality, para que las organizaciones vean el impacto en coste antes de la disponibilidad general. La función está en vista previa pública y muestra la métrica de committers que se usaría para facturar cuando Code Quality pase a ser de pago; GitHub aclara que la estimación no incluye minutos de Actions ni otros cargos por funciones AI. Esta herramienta permite planificar y decidir si mantener Code Quality habilitado antes del cambio de modelo. [GitHub Changelog: GitHub Code Quality license estimate in public preview](https://github.blog/changelog/2026-07-13-github-code-quality-license-estimate-in-public-preview/). ([github.blog](https://github.blog/changelog/2026-07-13-github-code-quality-license-estimate-in-public-preview/?utm_source=openai))

## GitHub Agentic Workflows: actualización semanal y runtimes sandbox
El proyecto GitHub Agentic Workflows publicó su actualización semanal donde anuncian la versión `v0.82.8` (publicada el 11 de julio) con varias mejoras: soporte de runtime `gVisor` para ejecutar agentes en sandbox, opción `docker-sbx` para aislamiento por VM con KVM, y correcciones a fallos de autenticación con Docker en flujos `sbx`. También destacan utilidades operativas como el agente `aw-failure-investigator` (agente on‑call que detecta fallos y abre issues), y mejoras en la reusabilidad de partials y manejo de tokens OAuth en conclusiones de jobs. Estas mejoras facilitan ejecutar agentes con mayor aislamiento y robustez en pipelines. [GitHub Agentic Workflows – Weekly Update (13 Jul 2026)](https://github.github.com/gh-aw/blog/2026-07-13-weekly-update/). ([github.github.com](https://github.github.com/gh-aw/blog/2026-07-13-weekly-update/))

## Go / gopls v0.23.0: compatibilidad con Go 1.27 y mejoras de análisis
El equipo de Go publicó la nota de `gopls` v0.23.0 (julio 2026), centrada en asegurar compatibilidad con las características de Go 1.27 RC y en correcciones y mejoras de análisis. Entre los puntos destacados están: sugerencias de autocompletado e inlay hints para campos promovidos en Go 1.27, nuevos analizadores (por ejemplo `sqlrowserr` para detectar llamadas a `Rows.Next()` sin comprobar `Err()`), ajustes en transformaciones de código (`stubmethods`, `fillstruct`) para las novedades de versiones recientes, y limpieza de comandos CLI obsoletos del binario `gopls`. Recomendado para desarrolladores que quieran soporte IDE actualizado para Go 1.27. [Gopls release v0.23.0 (Go.dev)](https://go.dev/gopls/release/v0.23.0). ([go.dev](https://go.dev/gopls/release/v0.23.0))

## GitHub: separación de SSO y Organizations en la configuración
GitHub publicó en su changelog una mejora de experiencia que separa la antigua página combinada de SSO y Organizations en dos páginas distintas dentro de los ajustes de usuario. El cambio busca facilitar la gestión de SSO y de organizaciones por separado, simplificando la navegación para administradores y usuarios con SSO u organización. Es una actualización menor pero de utilidad para equipos que manejan cuentas y accesos a nivel organizacional. [GitHub Changelog: Separate SSO and Organizations pages in Settings (13 Jul 2026)](https://github.blog/changelog/2026-07-13-separate-sso-and-organizations-pages-in-settings/).
