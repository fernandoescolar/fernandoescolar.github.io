## Google Cloud: resumen "What's new" (1 Jul 2026)

## Google Cloud: cambios de precio en Cloud Storage
Google Cloud publicó una nota con cambios de precio y nuevas SKUs para Cloud Storage que afectan, entre otros, la replicación y las transferencias de datos multi‑región. Parte del cambio introduce cargos por replicación y por transferencias entre regiones/multi‑regiones donde antes algunos movimientos eran gratuitos; también hay mapeos de SKUs antiguos a SKUs nuevos y ajustes por región. Estos cambios pueden impactar la factura de arquitecturas que dependen de replicación y acceso multi‑región y conviene revisar la tabla de SKUs y calcular el efecto en cargas de trabajo intensivas en datos.

## Azure: Chaos Studio Workspaces en preview público
Microsoft anunció que Azure Chaos Studio Workspaces está disponible en preview público. Workspaces ofrece escenarios predefinidos para probar la resiliencia de aplicaciones (por ejemplo, "Zone Down", fallos de DNS, failover de bases de datos) y un diseñador de escenarios que facilita componer fallos reales a nivel de infraestructura y aplicación. La propuesta incluye generación de informes estructurados tras cada prueba, integraciones con workflows (Skill para GitHub Copilot y control vía MCP) y orientación para aplicar estas pruebas a cargas de trabajo de IA y agentes en producción. Es una herramienta para validar la recuperación y detectar gaps antes de que ocurran incidentes reales.  
[Fuente: Microsoft Azure Blog — Proving application resilience on Azure with Chaos Studio (1 Jul 2026)].(https://azure.microsoft.com/en-us/blog/proving-application-resilience-on-azure-with-chaos-studio/) ([azure.microsoft.com](https://azure.microsoft.com/en-us/blog/proving-application-resilience-on-azure-with-chaos-studio/?utm_source=openai))

## Microsoft: WSL Containers en vista previa pública para ejecutar contenedores Linux en Windows
Microsoft puso en preview público su iniciativa de "WSL Containers" (herramienta `wslc.exe` y API), que permite crear, ejecutar y gestionar contenedores Linux directamente desde WSL sin depender de terceros. La funcionalidad incluye una CLI familiar y una API para integrar contenedores Linux desde aplicaciones Windows, controles de administración (por Intune) y mejoras de rendimiento como `virtiofs` para acceso a ficheros. Esto reduce la barrera para flujos de trabajo cloud‑native y ML locales en máquinas Windows y facilita, por ejemplo, ejecución de contenedores con acceso a GPUs en entornos de desarrollo.  
[Fuente: Windows Developer Blog / anuncios sobre WSL Containers (public preview)].(https://blogs.windows.com/windowsdeveloper/2026/06/02/build-2026-furthering-windows-as-the-trusted-platform-for-development/) ([blogs.windows.com](https://blogs.windows.com/windowsdeveloper/2026/06/02/build-2026-furthering-windows-as-the-trusted-platform-for-development/?utm_source=openai))

## containerd: CVE-2026-50195 — riesgo de envenenamiento del cache de imágenes vía checkpoint imports
Se publicó CVE‑2026‑50195, que afecta a containerd en su proceso CRI de importación de checkpoints: containerd no validaba referencias de imagen en la configuración de checkpoints, permitiendo que un atacante con permiso para crear pods forzase la descarga de una imagen maliciosa y la etiquetase localmente, "envenenando" el cache de imágenes del nodo. Si otros pods usan etiquetas locales con políticas `IfNotPresent`/`Never`, podrían ejecutar la imagen maliciosa. La NVD y el advisory de containerd indican versiones corregidas (p. ej. 2.3.2, 2.2.5, 2.1.9) — revisar e instalar las versiones parcheadas o mitigaciones del proveedor.  
[Fuente: NVD / GitHub security advisory — CVE-2026-50195].(https://nvd.nist.gov/vuln/detail/CVE-2026-50195) ([nvd.nist.gov](https://nvd.nist.gov/vuln/detail/CVE-2026-50195?utm_source=openai))

## NVIDIA Container Toolkit: CVE-2026-24260 (TOCTOU) — vulnerabilidad crítica en toolkit para Linux
NVIDIA publicó avisos que aparecen en el registro de seguridad sobre varias vulnerabilidades; entre ellas CVE‑2026‑24260, una condición de carrera TOCTOU en NVIDIA Container Toolkit para Linux que puede permitir ejecución remota de código, escalado de privilegios y manipulación de datos. El advisory oficial de producto‑security de NVIDIA y los registros NVD/CVE recomiendan actualizar el toolkit y el GPU Operator afectados. Este aviso es especialmente relevante para homelabs y clústeres que exponen GPUs en contenedores (por ejemplo, setups con `nvidia-docker`, GPU Operator o NVIDIA Container Toolkit).  
[Fuente: NVIDIA Product Security / NVD — Security bulletin y CVE-2026-24260].(https://github.com/NVIDIA/product-security/tree/main/2026/5850) ([vulners.com](https://vulners.com/vulnrichment/VULNRICHMENT%3ACVE-2026-24260?utm_source=openai))

## Ecosistema Go: Pion DTLS — CVE-2026-54908 (DoS) y corrección disponible
En el ecosistema Go se publicó CVE‑2026‑54908 que afecta a la implementación Pion DTLS: versiones anteriores a la 3.1.4 pueden sufrir un panic (DoS) al procesar un ServerKeyExchange ECDHE_PSK especialmente crafted. Los mantenedores han publicado la corrección (actualizar a la versión afectada o superior) y los equipos que usan `pion/dtls` en servicios expuestos deberían planear la actualización o mitigación cuanto antes. Este tipo de fallos es relevante para proyectos Go que implementan DTLS en gateways, dispositivos o infraestructuras de medios en tiempo real.  
[Fuente: GitHub / avisos CVE y correcciones — Pion DTLS (CVE-2026-54908)].(https://github.com/pion/dtls/security/advisories/GHSA-wg4g-wm44-ch5j) ([vulners.com](https://vulners.com/cve/CVE-2026-54908?utm_source=openai))

## Contenedores & seguridad: panorama de múltiples avisos y parches relevantes para homelabs y clusters
En las últimas 48 horas han convergido varios avisos y parches que afectan a ecosistemas de contenedores y GPU (containerd, NVIDIA Container Toolkit, actualizaciones de Docker/Engine y múltiples CVEs del kernel y componentes relacionados). Para administradores de homelab y clusters productivos esto significa: (1) priorizar actualizaciones de runtimes y toolkits GPU, (2) revisar políticas de pull (evitar `IfNotPresent` para tags inseguros), (3) auditar su cadena de images y CI/CD para referencias a checkpoints no confiables, y (4) programar ventanas de mantenimiento para aplicar parches del kernel y agentes de GPU. Mantener mirrors y backups de configuración, y probar actualizaciones en entornos aislados sigue siendo la mejor práctica.
